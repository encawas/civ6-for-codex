# 文明 6 自动化接口与转换层：想法整理

> 本目录目前用于整理新一代《文明 6》自动化工具的核心想法。
>
> 这不是完整实施计划，也暂不决定最终模块划分、开发顺序或工具数量。当前重点是先统一我们对“游戏接口、AI、转换器和验证”的理解，并据此继续调查《文明 6》能够读取和写入的实际能力。

## 一、核心想法

整个系统应当围绕一条固定边界建立：

> **AI 只负责根据游戏信息做自然语言层面的分析和判断，并输出符合固定格式的决策；AI 不直接生成 Lua、不直接调用文明 6 接口，也不负责确认操作是否成功。**

真正进入游戏的所有读取和写入，都由确定性的转换层完成。

可以把系统理解成双向翻译：

1. **读取方向**：把文明 6 的原始对象、状态和事件，翻译成稳定、统一、便于程序与 AI 理解的结构化信息。
2. **写入方向**：把上层的标准工具指令，翻译成文明 6 实际能够执行的 Lua 和原生游戏操作。
3. **验证方向**：写入后重新读取游戏事实，用新的游戏观测判断操作是否真正发生，而不是相信一次函数调用的返回值。

## 二、完整信息流

```text
文明 6 原始状态
        ↓
游戏读取层
        ↓
状态转换器：原始游戏对象 → 统一结构化状态
        ↓
AI：阅读状态，进行自然语言分析和决策
        ↓
规范化输出：固定动作名称 + 固定参数 + 自然语言理由
        ↓
输出校验器：检查格式、参数、权限和当前前置条件
        ↓
指令转换器：标准动作 → 文明 6 Lua / 原生游戏指令
        ↓
文明 6 执行请求
        ↓
记录动作尝试并结束本轮写入
        ↓
后续通过新的游戏读取结果进行验证
```

这条链路中，AI 和游戏写接口之间必须始终存在校验器与转换器，不能绕过。

## 三、AI 的职责

AI 负责回答的是：

- 当前局势意味着什么；
- 哪个目标更值得追求；
- 应选择哪个已有的标准动作；
- 为什么做这个选择；
- 在信息不足时，下一步需要补充读取什么信息。

例如，AI 可以判断：

> 东北方向仍未探索，勇士当前没有战斗任务，应向目标地块移动以扩大视野。

但 AI 输出给程序的可执行部分必须是固定格式，例如：

```json
{
  "analysis": "东北方向仍未探索，勇士当前没有战斗任务。",
  "decision": {
    "action": "move_unit",
    "arguments": {
      "unit_id": 12,
      "target_x": 25,
      "target_y": 31
    }
  }
}
```

其中：

- `analysis` 是自然语言判断，主要用于理解、审查和调试；
- `action` 必须来自程序事先定义的动作集合；
- `arguments` 必须符合该动作的固定参数结构；
- 只有 `decision` 可以进入后续校验流程；
- AI 不能把任意文字、Lua 代码或临时发明的动作直接送入游戏。

## 四、转换器是什么

这里的“转换器”不是简单的字符串替换器，而是文明 6 的确定性适配层。

对于每一种标准动作，它应明确知道：

- 需要哪些参数；
- 参数的类型和取值范围；
- 应在哪个 Lua 运行环境中执行；
- 应调用文明 6 的哪个原生接口；
- 执行前需要检查哪些条件；
- 指令已经发出时应记录什么；
- 后续需要重新读取哪些状态才能确认结果；
- 可能出现哪些明确拒绝和不确定结果。

例如，标准动作：

```json
{
  "action": "move_unit",
  "arguments": {
    "unit_id": 12,
    "target_x": 25,
    "target_y": 31
  }
}
```

转换器负责将其转换成固定、经过测试的文明 6 操作，大意类似：

```lua
local unit = Players[playerID]:GetUnits():FindID(12)
local params = {}
params[UnitOperationTypes.PARAM_X] = 25
params[UnitOperationTypes.PARAM_Y] = 31
UnitManager.RequestOperation(unit, UnitOperationTypes.MOVE_TO, params)
```

转换器不判断“这个方向是否值得探索”，也不根据自然语言临时猜测接口。它只负责把一个已经通过校验的标准动作，可靠地翻译成游戏指令。

## 五、读取与写入需要分别调查

重新开发前，需要逐步建立一份《文明 6》接口能力认知，而不是只研究当前已经用到的几个工具。

### 1. 读取能力

需要调查游戏能否稳定读取，以及应如何统一表达的内容，包括但不限于：

- 当前游戏、玩家、回合和游戏阶段；
- 当前阻止回合结束的事项；
- 单位的位置、类型、生命、移动力、状态和可执行操作；
- 城市人口、产出、区域、生产队列和可选生产项目；
- 地图地块、地形、资源、所有权、可见性和地块上的对象；
- 科技、市政、尤里卡和鼓舞状态；
- 外交关系、战争、开放边境、交易和外交会话；
- 政府、政策卡、总督、宗教、伟人等系统；
- 游戏事件和前后状态变化。

每一种读取能力至少需要整理：

```text
我们希望得到的信息
文明 6 的真实数据来源
需要使用的 Lua 环境
原始返回形态
统一后的结构化形态
已知限制与异常
信息是否可能过期
```

读取层只提供游戏事实，不夹带战略结论。

### 2. 写入能力

需要调查文明 6 能够稳定执行的操作，包括但不限于：

- 单位移动、攻击、驻防、休眠、升级、晋升和建设；
- 建城与其他特殊单位操作；
- 设置城市生产、购买、区域和奇观放置；
- 设置科技和市政研究；
- 外交、交易、政府、政策、总督和宗教操作；
- 结束回合、保存和读取游戏。

每一种写入能力至少需要整理：

```text
标准工具动作名称
输入参数
执行前条件
文明 6 原生接口
使用的 Lua 环境
请求发出后的直接反馈
后续需要重新读取的验证信息
明确拒绝条件
结果不确定时的处理边界
```

这份能力整理将成为后续转换器的依据。

## 六、写入请求与执行成功必须分开

文明 6 接受一个操作请求，并不代表操作已经真正完成。

可能出现：

- 请求没有报错，但游戏没有执行；
- 操作被弹窗或当前界面状态拦截；
- 单位只移动了一部分路径；
- 游戏正在处理其他逻辑；
- 操作已经成功，但直接返回没有说明；
- 连接中断，无法确认请求是否生效。

因此系统必须区分：

1. **前置检查结果**：当前是否允许尝试发送该动作；
2. **动作尝试**：具体指令是否已经送入游戏；
3. **验证结果**：后续新的游戏观测能否证明动作成功、失败或仍不确定。

建议统一表达的结果概念包括：

- `precondition_failed`：未发送，前置条件不满足；
- `dispatched`：指令已经发出，但尚未通过新观测确认；
- `confirmed`：后续新观测证明动作成功；
- `rejected`：游戏明确拒绝，或新观测证明没有发生且原因明确；
- `uncertain`：指令可能已经产生效果，但当前证据不足，不能盲目重试。

结果判断由规则代码和真实游戏观测完成，不能由 AI 自己声称成功。

## 七、各模块的固定职责

| 模块 | 负责 | 不负责 |
|---|---|---|
| 游戏读取层 | 读取文明 6 原始事实 | 不做战略分析 |
| 状态转换器 | 将原始事实整理成统一结构 | 不改变或推测事实 |
| AI | 分析局势、选择标准动作、给出理由 | 不生成 Lua、不直接操作游戏 |
| 输出校验器 | 校验动作名称、参数、权限和前置条件 | 不判断战略是否优秀 |
| 指令转换器 | 将标准动作翻译成固定游戏指令 | 不自由发挥、不做战略决策 |
| 动作记录层 | 记录已经发出的写入尝试 | 不把“已发送”当成“成功” |
| 结果验证器 | 根据后续新观测确认实际结果 | 不依赖 AI 的主观判断 |

## 八、为什么这条路线效率更高

这种设计的主要价值不是减少某一次调用的代码量，而是避免每一层都重复理解文明 6。

只要游戏适配层稳定，上层可以自由替换或增加：

- 不同大模型；
- 纯规则程序；
- MCP（模型上下文协议）工具；
- 网页控制台；
- 自动测试程序；
- 批量跑局和评估系统。

这些上层系统只需要理解统一的结构化状态与标准动作，不需要重新学习 FireTuner、Lua 环境和文明 6 的内部接口差异。

因此，最重要的工程资产不是某个提示词或某个智能体，而是：

> **一套经过调查、测试和验证的文明 6 读取接口、标准动作契约与双向转换层。**
