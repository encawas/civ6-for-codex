# Civilization VI Workflow Runtime Refactor Report

报告日期：2026-07-15
基线分支：`main`
上游基线：`5e61237` (`docs: add Phase 0 runtime audit and test catalog`)
当前阶段：Phase 1 - Canonical Domain Contracts

## 1. 执行摘要

本轮重构完成了规范领域层的第一批落地，为后续替换旧运行时建立了稳定的数据语义和结构性安全约束。

本轮没有直接删除旧引擎、`safe_*` 模块或导入时替换逻辑。仓库的重构章程明确禁止一次性清空重写，要求先通过特征测试冻结行为，再逐边界切换并删除旧实现。因此，本轮采用“新领域契约先落地、旧执行路径保持不变”的迁移方式。

完整测试结果：`113 passed`，其中原有回归测试 92 项，新增领域测试 21 项。

## 2. 重构目标

本轮针对以下问题建立可执行的领域约束：

- 旧模型混合观察、事件、计划、任务和执行状态；
- 任务只有调用方生成的 `task_id`，缺少语义幂等身份；
- 游戏槽位缺少统一的空值表达和所有权冲突判断；
- 动作发送、工具返回和游戏状态验证容易被错误合并为一次“成功”；
- 单 Tick 最多一次游戏变更尚未成为可验证的数据约束；
- 审批、规划请求、动作尝试等记录缺少独立生命周期。

## 3. 已完成内容

### 3.1 规范领域模型

新增 `src/civ6_workflow/domain/`，分别定义：

| 领域概念 | 模块 | 关键约束 |
|---|---|---|
| Observation | `observations.py` | 独立观察身份、序号、来源版本和稳定投影哈希 |
| Event | `events.py` | 独立事件生命周期和解析证据 |
| Decision gap | `decisions.py` | 战略决策缺口与系统故障分离 |
| Plan | `plans.py` | 修订号、有效期、完成条件和失效条件 |
| Task | `tasks.py` | 语义幂等键、槽位、执行窗口和审批状态 |
| Action attempt | `attempts.py` | `PREPARED`、`VERIFYING`、`UNCERTAIN` 等独立状态 |
| Approval | `approvals.py` | 面向精确提案修订的不可变审批记录 |
| Planner request | `planner.py` | 逻辑请求与底层 provider 尝试分离 |
| Workflow Tick | `ticks.py` | 闭合结果类型和 0/1 游戏变更预算 |

所有规范模型使用严格字段校验、禁止额外字段，并冻结顶层实例，降低非法状态组合进入运行时的概率。

### 3.2 观察归一化

新增统一槽位状态：

- `EMPTY`
- `OCCUPIED`
- `NOT_LOADED`

以下上游空值现在映射为同一个规范值：

```text
None, "", "NONE", "none", "nothing", "NOTHING", {}, []
```

这为后续把协议拼写判断移出核心规则提供了单一边界。

### 3.3 任务幂等与槽位冲突

任务幂等键由以下语义字段稳定生成：

- 游戏会话；
- 主体类型和主体 ID；
- 规范槽位；
- 期望结果；
- 来源计划修订；
- 执行窗口。

调用方即使使用不同 `task_id`，也不能绕过相同语义工作的重复检测。处于活动状态且执行窗口重叠的任务不能同时占用同一槽位。

### 3.4 动作和 Tick 安全表达

动作尝试现在能够区分：

- 已准备但尚未发送；
- 已发送且等待验证；
- 发送结果不确定；
- 后续观察确认成功；
- 明确失败或发送前拒绝。

`WorkflowTick` 的变更预算被限制为 `0` 或 `1`。发送普通动作和 `end_turn` 都必须消耗唯一一次预算。

## 4. 测试覆盖

新增测试文件：`tests/test_canonical_domain.py`。

已覆盖重构测试目录中的稳定编号：

| 编号 | 结果 | 覆盖内容 |
|---|---|---|
| `OBS-001` | PASS | 所有空生产拼写归一化一致 |
| `OBS-002` | PASS | 有效生产项目保持非空 |
| `OBS-006` | PASS | 相同事实拥有不同观察 ID 和相同投影哈希 |
| `TASK-001` | PASS | 重复语义生成稳定幂等身份 |
| `TASK-002` | PASS | 不同任务 ID 不能绕过语义冲突 |
| `TASK-003` | PASS | 重叠执行窗口的槽位冲突在调度前被发现 |

额外覆盖：

- 冻结模型禁止顶层字段修改；
- 伪造幂等键被拒绝；
- 未发送的动作不能进入验证状态；
- 单 Tick 第二次变更被结构性拒绝；
- 规范任务支持 JSON 无损往返。

| 范围 | 本轮影响 |

| SQLite schema | 无变化，无数据迁移 |
| Planner 调用 | 无变化，调用资格和预算尚未切换 |

本轮回滚不涉及数据库：删除 `civ6_workflow.domain`、新增测试和两份进度文档即可恢复到上游基线。

## 6. 尚未完成的工作

当前仍存在以下旧架构问题：

- `civ6_workflow.__init__` 仍执行类、注册表和实现替换；
- `safe_*` 模块和多层引擎继承仍是有效运行路径；
- 旧 Tick 仍可能创建并执行多个任务；
- 动作发送和后续新鲜观察验证尚未完全分离；
- 规范模型尚未写入 SQLite；
- 规划资格仍由旧事件路由和每回合布尔记录控制；
- 控制台尚未展示规范 Tick、动作尝试和决策缺口。

因此，本报告不宣称运行时重构已经完成。

## 7. 下一阶段建议

下一阶段应保持一次只迁移一个边界：

1. 定义规范 Game、Planner、Store 和 Clock 端口；
2. 增加显式 `bootstrap.py` 组合根；
3. 将旧快照翻译为规范 Observation；
4. 通过对比测试切换观察归一化边界；
5. 建立有界 Tick runner 和记录型 Game port；
6. 让任务物化结束当前 Tick；
7. 再迁移动作尝试持久化和后续 Tick 验证；
8. 每完成一个切换边界，删除对应旧实现，不保留双重权威。

## 8. 验收结论

本轮满足 Phase 1 第一批领域契约的验收目标：

- 核心概念拥有独立、严格的规范类型；
- 空槽、幂等身份和槽位冲突具有单一语义；
- 关键非法状态在模型构造阶段被拒绝；
- 新增测试与原有回归测试全部通过；
- 未改变旧运行时、持久化或游戏操作行为。

该结果可以作为下一轮观察边界和有界 Tick 状态机迁移的稳定基础。
